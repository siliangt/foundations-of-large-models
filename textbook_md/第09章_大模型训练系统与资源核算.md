# 第9章 大模型训练系统与资源核算

在上一章我们了解到，大模型的参数量动辄几十亿、几百亿甚至上千亿。然而，一张顶级的英伟达 GPU（如 A100 或 H100）的显存通常只有 80GB。如果我们试图把一个 1750 亿参数的 GPT-3 塞进一张单卡里，它甚至连模型本身的参数都装不下，更别提还需要保存梯度和优化器状态了。

当算法的野心超出了单张芯片的物理极限时，我们就必须借助**大规模分布式训练系统**。本章将详细探讨大模型训练所面临的“算力墙”与“显存墙”，以及工业界是如何通过 3D 并行和 ZeRO 优化技术，将成百上千张 GPU 联结成一台超级计算机的。

## 9.1 算力墙与显存墙：大模型训练的物理极限

在训练大模型时，我们主要面临两大瓶颈：
1. **显存墙（Memory Wall）**：训练一个模型所需的显存，远远不止模型参数那么简单。以 Adam 优化器为例，它需要为每个参数保存一阶动量和二阶动量。在半精度（FP16/BF16）训练下，10 亿参数（1B）通常需要消耗约 16GB 到 20GB 的显存。这意味着，一个百亿级别的模型直接让单张 80G 显卡爆显存。
2. **算力墙（Compute Wall）**：即使假设我们有无限大的显存，计算上千亿参数前向和反向传播的矩阵乘法，单张卡可能需要运行几十年。

为了打破这两堵墙，我们必须将计算和数据切分到多张卡上，这就是**并行训练（Parallel Training）**。

## 9.2 数据并行（DP）与分布式训练基础

最简单、最基础的并行策略是**数据并行（Data Parallelism, DP）**。

### 9.2.1 DP 的原理
在数据并行中，**每张 GPU 上都复制一份完整的模型副本**。
我们将庞大的训练数据（Batch）切分成多个小的数据块（Micro-batch），分发给不同的 GPU。每张卡独立计算前向传播和自己那部分数据的梯度。
在反向传播结束后，所有 GPU 通过集合通信（All-Reduce 操作）将各自算出的梯度进行同步汇总，求出平均梯度，最后所有卡同步更新自己的模型参数。

### 9.2.2 DP 的局限性
数据并行非常容易实现，且并行效率极高。但它的致命缺点是：它要求单张 GPU 必须能够完整装下整个模型及训练状态。当模型太大，单卡装不下时，纯粹的数据并行就失效了。

## 9.3 突破单卡极限的 3D 并行策略

为了切分模型本身，大模型训练引入了著名的 **3D 并行**：数据并行（DP）、张量模型并行（TP）和流水线模型并行（PP）。

### 9.3.1 张量模型并行（Tensor Parallelism, TP）
张量并行是对算子层面的“微观”切分（代表框架：Megatron-LM）。
以矩阵乘法 $Y = XA$ 为例，我们可以把巨大的参数矩阵 $A$ 沿着列或行切分成几块，分别放到不同的 GPU 上。计算时，各张 GPU 计算自己负责的那部分乘法，然后再通过网络通信（All-Gather 或 Reduce-Scatter）拼成完整的结果。
- **优点**：能够完美解决单卡装不下大算子的问题。
- **缺点**：每次矩阵乘法都需要频繁的网络通信，对 GPU 之间的带宽要求极高。因此，TP 通常只在同一台机器（Node）内部、通过极高速的 NVLink 连接的几张卡之间使用，很少跨机器使用。

### 9.3.2 流水线模型并行（Pipeline Parallelism, PP）
流水线并行是对网络层面的“宏观”切分。
假设我们有一个 24 层的 Transformer，我们可以把第 1-8 层放在 GPU 0 上，9-16 层放在 GPU 1 上，17-24 层放在 GPU 2 上。前向传播时，数据像流水线一样依次经过 GPU 0 -> GPU 1 -> GPU 2。
- **优点**：机器之间的通信量很小（只需要传递激活值），适合跨机器、跨节点的大规模集群使用。
- **缺点**：存在“气泡（Bubble）”时间。当 GPU 0 在计算时，GPU 1 和 2 只能干等。虽然工业界提出了 Micro-batch 调度机制（如 1F1B 算法）来填补气泡，但计算资源的浪费始终存在。

## 9.4 零冗余优化器（ZeRO）：内存管理的魔法

虽然 3D 并行极其强大，但代码实现异常复杂。2020 年，微软提出了惊艳的 **ZeRO（Zero Redundancy Optimizer）** 技术（DeepSpeed 框架的核心），它为数据并行带来了一场革命。

在传统的数据并行中，由于每张卡都保存了完整的优化器状态、梯度和参数，这造成了极大的**冗余**。ZeRO 的核心思想是：**消除这些冗余！**

ZeRO 分为三个阶段：
- **ZeRO-1**：将优化器状态（占显存最大的部分）切分到各张卡上。
- **ZeRO-2**：进一步将梯度也切分到各张卡上。
- **ZeRO-3**：将模型参数本身也切分到各张卡上。

在 ZeRO-3 状态下，单张卡平时只保存属于自己的一小部分模型参数。当它在计算某一层的算子时，如果发现缺了其他参数，就通过网络向其他卡“借（Broadcast/All-Gather）”过来，算完马上“扔掉”。
**ZeRO 以极其优雅的方式，用通信换取显存，让数据并行（FSDP）能够训练千亿级别的大模型，极大地降低了分布式训练的门槛。**

## 9.5 显存卸载（Offload）技术

除了榨干多张 GPU 的显存，研究人员还盯上了相对廉价的 CPU 内存和 NVMe 硬盘。
**Offload 技术**指的是：在模型训练时，把暂时用不到的优化器状态或参数转移到 CPU 内存上保存；等需要更新时，再把它们传回 GPU。这使得我们能够在有限的硬件（甚至单张消费级显卡）上，微调原本根本装不下的大模型。虽然这会大幅牺牲训练速度，但为平民化的大模型研究提供了可能。

在掌握了如何动用成百上千张 GPU 训练出基座模型后，下一个面临的问题是：基座模型太贵、且不符合具体的垂直领域需求。在下一章，我们将探讨如何以极低的成本，通过微调技术让庞大的模型“为你所用”。
